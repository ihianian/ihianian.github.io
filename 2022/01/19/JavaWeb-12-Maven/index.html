1. Maven简介

Maven是一个项目管理工具，是一个辅助工具，不用它也可以完成，只不过比较繁琐。那么说到工具，就会想到Git，也是一个项目管理工具，负责版本控制。那么Maven主要是干什么的呢？

我们知道，一个完整的项目，比如JavaWeb项目，除了编写代码之外，还需要一些配置文件，比如Jar包、xml文件等，以及各个文件之间的调用等等。这里回顾一下软件工程的概念：为了能够实现软件的流水线式生产，在设计和构建软件时能够有一种规范和工程化的方法，人们便提出了软件工程概念。

而在项目开发过程中，需要做的工具如下所示：

  1. 分析项目要做什么，功能有哪些？
  2. 设计项目，步骤、技术、人力、时间等等。
  3. 组件团队，构建设备，软件以及硬件。
  4. 开发人员写代码，测试程序，调Bug。
  5. 测试人员对项目功能进行测试，是否符合功能要求。有问题则返回给开发人员，重复4、5步骤直到测试成功。

那么在开发过程中存在哪些问题呢？

  1. 模块很多，之间的关系复杂，手动管理模块比较繁琐；
  2. 项目需要很多的第三方库，Jar包，需要手工从网络中获取Jar包；
  3. Jar包的版本需要手动管理，版本需要匹配；
  4. 管理Jar包之间的依赖，比如a.jar包需要b.jar包中的类，这时候就需要手动将a和b都要下载到项目中，a需要b这个关系就叫做依赖。

1.1 项目管理

上述的操作很繁琐耗时，这时候就需要Maven改进项目的开发和管理。

  • Maven可以管理jar文件；
  • 自动下载jar和其他的文档、源代码；
  • 管理jar直接的依赖，比如a.jar需要b.jar，那么就会自动下载b.jar包；
  • 管理需要的jar版本；
  • 编译程序，将.java编译为.class，测试代码是否正确；
  • 帮助打包文件，形成jar文件或者war文件；
  • 帮助部署项目。

1.2 项目构建

另外，编译、打包以及测试这些过程也被称为构建的过程。项目构建是面向过程的，就是一些步骤，完成项目代码的编译、测试、运行、打包、部署等等。Maven支持的构建有：

  1. 清理，即将之前项目编译的东西删除掉，为新的编译代码做准备；
  2. 编译，即将程序源代码编译为可执行代码，这里的编译是批量的，可以同时把成百上千的文件编译为class文件；
  3. 测试，Maven可以执行测试程序代码，验证功能是否正确，也是可以批量执行的；
  4. 报告，生成测试结果的文件，比如测试是否通过；
  5. 打包，将项目中的所有class文件、配置文件等等打包到一个压缩文件jar包（普通Java程序）或者war包（JavaWeb应用程序）；
  6. 安装，将jar包或者war包安装到本机仓库；
  7. 部署，把程序安装好可以执行。（用的少，主要是前6步）

简单地说，Maven的作用就是：

  1. 管理依赖：Jar包的管理、下载和版本控制；
  2. 构建项目：完成项目代码的编译、测试、打包和部署；

2. Maven的核心概念

首先介绍一下Maven中的名词概念，如下所示：

  1. POM

    POM即项目对象模型（Project Object Model），Maven把一个项目当做一个模型来使用。与之关联的文件是pom.xml，和Servlet中的web.xml类似，对应整个项目，对项目的上述所有控制都是通过pom.xml文件来实现的。

  2. 约定的目录结构

    Maven项目的目录和文件的位置都是有规定的。注意，这是约定，是大家都要遵循的规则。不是强制的，但是比规范要强势一点。

  3. 坐标

    是一个唯一的字符串，用来表示资源的。

  4. 依赖管理

    管理项目所使用的jar包的。

  5. 仓库管理（了解）

    资源存放的位置。

  6. 生命周期（了解）

    Maven工具构建项目的过程，就是生命周期，即上面7个过程。

  7. 插件和目标（了解）

    执行Maven构建的时候所用的工具就是插件，比如打包所用的工具等等；目标就是想干什么，比如打包。

  8. 继承

    和Java中的继承类似，见后续。

  9. 聚合

    将好多个项目聚合到一起，见后续。

2.2 仓库管理

前面简单提到了中央仓库和本地仓库两个概念。

那么仓库是什么呢？仓库是存放东西的，比如存放maven使用的jar包以及项目使用的第三方jar包。

仓库主要分为本地仓库和远程仓库两类：

  • 本地仓库，就是在个人计算机上的文件夹，存放各种jar。
  • 远程仓库，就是在互联网上的仓库，使用网络可以访问并下载上面的资源。远程仓库又分为下面三类仓库：
    • 中央仓库：最权威的远程仓库，所有的开发人员都共享使用的一个集中的仓库；
    • 中央仓库镜像：相当于中央仓库的镜像，在各大洲以及各大城市都有镜像，方便局域内的人员快速访问。
    • 私有仓库（私服）：在公司内部局域网内使用的，不是对外使用。

这里Maven仓库的概念和Git以及Docker的仓库类似。

2.2.1 仓库的使用

仓库有那么多种类，那么Maven是怎么使用仓库的呢？实际上Maven仓库的使用不需要人为参与。

比如开发人员需要使用mysql驱动，

  1. Maven首先会检查本地仓库；
  2. 如果本地仓库没有，则会判断是否设置了私服，有的话，则会访问私服查找mysql驱动，私服内有的话，则会下载到本地仓库，然后再返回程序中；
  3. 如果没有设置私服或者私服中没有该jar包，则会访问中央仓库镜像，如果有驱动，则会下载到私服中，然后再从私服中下载到本地仓库中，最后返回程序中；
  4. 如果镜像也没有，那么就会访问中央仓库，有的话，则逐级下载到本地并返回给程序，中央仓库如果也没有那就真没有了。

也就是说，上述几类仓库是有优先级的，但是无论如何，本地仓库都会最终保留一份，方便下次使用。

2.2.2 修改本地仓库地址

前面提到，maven默认的本地仓库是用户目录下的.m2/repository下，但是这是系统盘，重装系统的话比较危险。可以在maven安装目录/conf/settings.xml配置文件中修改本地仓库，一般情况下建议提前备份一份。

在配置文件中找到localRepository标签，修改目录即可，注意不要有中文。

2.3 pom.xml文件（重点）

POM即Project Object Model项目对象模型。Maven把一个项目的结构和内容抽象成一个模型，在xml文件中进行生命，以方便进行构建和描述。pro.xml文件是Maven的灵魂，对该项目Maven操作的核心。作为xml文件，该文件由若干个标签组成。常用的标签如下所示：

标签 描述
基本信息
modelVersion Maven模型的版本，现在只能是4.0.0
groupId 组织名称，一般是公司域名倒写或者倒写+项目名
artifactId 项目名称，也是模块名称，对应groupId项目中的子项目
version 项目的版本号，如1.1.1。如果项目还在开发中，或者不稳定版本，通常后缀带-SNAPSHOT。
packaging 项目打包的类型，可以有jar、war、rar、ear、pom，默认是jar
依赖
dependencies 用于包含多个dependency依赖
dependency Maven的一个重要作用就是管理jar包。Maven通过该标签来配置，如下图，该标签内嵌套了坐标三个标签，形成一个唯一的依赖坐标。用于Maven检索仓库。相当于Java中的import
配置属性
properties 用来设置属性，例如项目构建源码编码方式，Java版本等等。
构建
build 项目构建时的相关配置，例如设置编译插件的JDK版本。
继承
parent 在Maven中，如果多个模块都需要声明相同的配置，例如：groupId、version、有相同的依赖、或者相同的组件配置等，也有类似 Java 的继承机制，用parent声明要继承的父工程的pom配置。
聚合
modules 在Maven的多模块开发中，为了统一构建整个项目的所有模块，可以提供一个额外的模块，该模块打包方式为pom，并且在其中使用modules聚合的其它模块，这样通过本模块就可以一键自动识别模块间的依赖关系来构建所有模块，叫Maven的聚合。

2.6 生命周期

对项目的构建是建立在生命周期模型上的，它明确定义项目生命周期各个阶段，并且对于每一个阶段提供相对应的命令，对开发者而言仅仅需要掌握一小堆的命令就可以完成项目各个阶段的构建工作。

构建项目时按照生命周期顺序构建，每一个阶段都有特定的插件来完成。不论现在要执行生命周期中的哪个阶段，都是从这个生命周期的最初阶段开始的。

对于我们程序员而言，无论我们要进行哪个阶段的构建，直接执行相应的命令即可，无需担心它前边阶段是否构建，Maven都会自动构建。这也就是Maven这种自动化构建工具给我们带来的好处。

2.7 Maven的常用命令

Maven对所有的功能都提供相对应的命令，要想知道maven都有哪些命令，那要看 maven 有哪些功能。maven主要有三大功能：管理依赖、构建项目、管理项目信息。

  • 管理依赖，只需要声明就可以自动到仓库下载；
  • 管理项目信息其实就是生成一个站点文档，一个命令就可以解决，最后再说；
  • maven 功能的主体其实就是项目构建。Maven提供一个项目构建的模型，把编译、测试、打包、部署等都对应成一个个的生命周期阶段，并对每一个阶段提供相应的命令，程序员只需要掌握一小堆命令，就可以完成项目的构建过程。

常用命令如下所示：

命令 描述
mvn clean 清除（会删除原来编译和测试的target目录，但是已经安装到本地仓库里的包不会删除）
mvn compile 编译主程序（会在项目主目录下生成一个target目录，里面存放编译主程序之后生成的字节码文件）
mvn test-compile 编译测试程序（会在项目主目录下生成一个target目录，里面存放编译测试程序之后生成的字节码文件）
mvn test 测试（会生成一个目录surefire-reports，保存测试结果）
mvn package（常用） 打包主程序（会编译、编译测试、测试，并且按照pom.xml配置把主程序打包成jar包或者war包）
mvn install（常用） 安装主程序（会把本工程打包，并且按照本工程的坐标保存到本地仓库中）
mvn deploy 部署主程序（会把本工程打包，按照本工程的坐标保存到本地仓库中，并且还会保存到私服仓库中，还会自动把项目部署到web容器中）

注意，以上命令必须在项目的根目录下运行，也就是在pom.xml所在目录下运行。

Maven的上述功能命令是通过插件来完成的，而Maven是由Java编写的，所以插件就是一段代码，也就是Jar包。

首先设置两个地方（就是告诉IDEA自己安装的maven在哪里）

  • File/settings，这个设置是对当前项目有效的；
  • File/Other Settings，这个设置是对以后其他所有项目有效的。

  • Maven home directory：maven的安装目录
  • User setting file：maven的配置文件
  • Local respository：本地仓库

4. 依赖管理

我们在添加依赖的时候，主要是有坐标三个标签，那么还剩下<scope>这个标签，这个标签表明该依赖的作用范围，就是在maven构建项目的哪些阶段中起作用。

  • test值，只在测试程序中起作用，比如junit.jar包，只在测试阶段用到，我们可以设置其作用范围为test，那么打包的时候就不会将该jar包打包进去，节省空间。
  • provided值，这个就是在程序员编写阶段起作用，比如servlet，不需要打包安装部署，因为项目实际的运行环境中包含了所需的jar包。可以看一下上述打包的war包，解压后lib目录中没有servlet、jsp等jar包。
  • compile值，在所有阶段都用到。

5. Maven常用设置

5.1 Maven的属性设置

属性设置是在pom.xml文件中的<properties>标签中进行设置，常用的属性有：

  • <project.build.sourceEncoding>，构建项目时的编码方式
  • <maven.compiler.source>，编译代码时的JDK版本
  • <maven.compiler.target>，运行程序时的JDK版本

5.2 Maven的全局变量

除了上述自带的属性，我们也可以自定义属性。在<properties>标签中定义属性（相当于全局变量），在其他位置中使用属性（相当于全局变量）。其中标签名就是属性名，标签嵌套的值就是属性值，后续使用的时候采用${标签名}的方式来利用属性值。比如spring中的很多依赖都是同一个版本号，那么可以将其抽象成变量，这样可以方便统一修改。

5.3 指定资源插件

默认情况下，maven执行代码的时候，会将src/main/resources目录中的资源文件拷贝到src/target/classes目录中，对于resources目录之外的其他非java文件不处理，所以这就要求我们将非java文件放到src/main/resources目录中。

但是有时候我们的程序需要把一些文件放到src/main/java目录中。我们可不可以设置一下也将该目录下的java文件拷贝呢？换句话说，我们可以可以额外指定将哪些目录下的资源文件拷贝呢？答案是可以的，在<build>中加入<resources>。

注意，这样的话相当于覆盖了Maven默认的设置，对于resources文件夹下不会再拷贝，此时再添加一个resources目录的拷贝即可。
